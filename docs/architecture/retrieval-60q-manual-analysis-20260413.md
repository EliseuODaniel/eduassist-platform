## Fairness Notes

This benchmark compares end-to-end user outcome under the same task, auth, and policy conditions, not identical internal implementation.

The most important fairness decisions in this round were:

- protected cases reused the same guardian Telegram identity across all stacks;
- public, protected, and restricted slices stayed mixed in the same battery;
- all four stacks ran with the same local Gemma baseline unless the stack had a deterministic shortcut for the specific case;
- the same dataset and rubric were reused across the four paths.

One practical caveat remains: the generated dataset file still reuses the historical filename `retrieval_50q_probe_cases.generated.json`, even when the battery size is `60`. The content of the run, however, really contains `60` prompts and the run prefix above is the correct source of truth.

## Slice-by-Slice Reading

### Public

`specialist_supervisor` was again the strongest public stack in this round.

- quality: `97.7`
- keyword pass: `30/32`
- average latency: `685.9 ms`

The public actionability work mattered here. Multi-document bundles such as permanence/support and health/second-call now produce more actionable sequencing instead of static summaries.

`python_functions` and `llamaindex` converged on public quality around `91`, which is a good sign for the shared semantic router and grounded public composer. `langgraph` stayed usable, but the four request failures that landed inside public slices kept it below the others.

### Protected

Protected performance converged even more tightly:

- `python_functions`: quality `95.3`
- `llamaindex`: quality `91.4`
- `langgraph`: quality `90.6`
- `specialist_supervisor`: quality `90.6`

This matters because the project no longer has a single stack monopolizing protected usefulness. The main issue now is not broad protected instability. It is narrower: follow-up phrasing around attendance and academic component pressure still loses specificity in some prompts.

### Restricted

Restricted internal-document behavior remains the biggest separator.

- `langgraph`: quality `96.4`
- `llamaindex`: quality `96.4`
- `specialist_supervisor`: quality `96.4`
- `python_functions`: quality `70.2`

This is the clearest unresolved gap left by this round. The trace calibration also reinforces the same conclusion: restricted retrieval still shows `answerable_rate=0.0%` in the trace analysis, which means the policy and telemetry are in place, but metadata/query variants/chunk selection are not yet mature enough to produce grounded wins consistently.

## Recurrent Failure Patterns

### 1. Restricted retrieval is still the main open problem

Evidence:

- trace recommendation explicitly calls out authenticated restricted retrieval with `answerable_rate=0.0%`;
- `python_functions` still lags sharply in the restricted slice;
- even the stronger stacks still solve some restricted prompts through safe no-match behavior rather than high-confidence grounded retrieval.

Why this matters:

This is not only a retrieval-quality issue. It directly affects the assistant's credibility in staff and internal-document flows, which are high-value and policy-sensitive.

### 2. Attendance follow-up phrasing still drifts

The architectural fixes helped the system keep attendance context more often, but some prompt variants still collapse into generic academic or conduct summaries instead of naming the more specific attendance pressure the rubric expects.

### 3. Public actionability improved, but unevenly

The new public multi-document answers are clearly better than in the previous round. Even so, `weak_actionability` still appears in `python_functions`, `llamaindex`, and once in `specialist_supervisor`.

This is important for the TCC because it separates:

- factual correctness
- from operational usefulness

The benchmark shows that those two properties correlate, but they are not identical.

### 4. Boundary handling for explicitly external entities still leaks

The prompt about the city public library exposed a residual problem across more than one stack. The system can still answer with the school library when the entity pivot should have stopped the school fast path entirely.

### 5. Residual request failures still dominate the worst cases

The biggest quality losses in `langgraph`, `python_functions`, and `llamaindex` came from `request_failed` exceptions, not from nuanced answer-style misses. This is a good sign for diagnosis: stability fixes should still yield immediate benchmark gains.

## Threats To Validity

1. Protected slices used a real guardian Telegram context.

This was the correct choice for fairness in this repository, but it means the benchmark reflects a realistic local serving setup, not a fully synthetic auth-only harness.

2. Local Gemma serving is not the same as cloud production serving.

Latency and some answer-style behavior remain shaped by the local `Gemma 4 E4B` runtime profile.

3. Keyword pass is still only one dimension.

Several answers with low keyword pass were directionally correct but not actionable enough. Conversely, some keyword-complete answers still had the wrong emphasis.

4. The trace report is still sparse for some stacks.

The telemetry improvements are strong enough to guide the next round, but trace coverage is still uneven across the four stacks, especially outside explicit retrieval-heavy paths.

## Practical Conclusions

1. The architectural program from this round worked.

The combination of:

- restricted retrieval tuning,
- attendance follow-up harmonization,
- and more actionable public grounded synthesis

produced measurable gains, especially in `llamaindex` and the overall public slice.

2. `specialist_supervisor` is the strongest general-purpose stack today.

It is the best overall candidate for product-facing quality under the current local setup.

3. `python_functions` is now strongest in protected deterministic service, but not in restricted retrieval.

This is an important nuance for design decisions: one stack can lead in protected structured data while still lagging in internal-document search.

4. `llamaindex` improved enough to stay strategically relevant.

The 60Q shows that the stack is no longer just a rescue path. It is now a credible participant in the comparison baseline.

5. The next round should be narrower and more empirical.

The repository now has enough telemetry and eval infrastructure to move from large structural refactors to tighter, capability-specific calibration.

## Recommended Next Priorities

1. Harden restricted retrieval for `python_functions`.
   Focus on metadata, query variants, chunk selection, and teacher-feedback longform prompts.
2. Fix residual request failures in `langgraph` and `llamaindex`.
   The biggest remaining losses are still exceptions.
3. Close the external-entity boundary leak across all stacks.
   The city-library prompt should be a stable out-of-scope or external-entity boundary everywhere.
4. Keep tightening attendance-detail phrasing.
   The stack should keep saying the right thing, but with the right emphasis for the rubric.
5. Preserve the 60Q battery as the new canonical local benchmark.
   This round is broad enough to serve as the next serious reference for the TCC draft.
