## Fairness Notes

This benchmark was intentionally configured to compare end-to-end user outcome under the same task, policy, and auth context, not to force the same internal implementation strategy.

The most important fairness decisions in this round were:

- protected cases used the same guardian auth surface across all four stacks;
- the specialist stack received the same guardian Telegram identity context it needs in real serving;
- public, protected, and restricted slices stayed mixed in the same battery;
- the same dataset and same rubric were reused for all stacks.

## Slice-by-Slice Reading

### Public

`specialist_supervisor` was the strongest stack in the public slice in this round.

- quality: `97.2`
- keyword pass: `26/29`
- average latency: `542.5 ms`

This matters because public questions are the most visible surface of the product, and the specialist stack is no longer paying a premium-latency penalty on the majority of those queries. The grounded public composer and the tighter fast-path/resolved-intent contracts clearly helped here.

`langgraph` and `llamaindex` were still usable in public retrieval, but both showed some loss on keyword precision and a few answers that stayed too generic.

`python_functions` remained solid, but its public answers were often a little less actionable than the specialist’s final wording.

### Protected

Protected performance converged more than public performance.

- `langgraph`: quality `92.0`
- `python_functions`: quality `89.2`
- `llamaindex`: quality `89.2`
- `specialist_supervisor`: quality `88.2`

The main conclusion here is not that one stack dominated protected data. The more relevant finding is that the recent routing and follow-up fixes removed the previous structural instability. All four stacks are now materially serviceable in protected flows, but they still differ in how often they over-clarify or lose the intended metric.

### Restricted

Restricted internal-document handling is still the biggest separator between stacks.

- `specialist_supervisor`: quality `95.6`
- `langgraph`: quality `95.6`
- `python_functions`: quality `71.1`
- `llamaindex`: quality `44.4`

This is the clearest unresolved gap in the current system. The automated trace analysis also pointed in the same direction: restricted retrieval remains the main place where retrieval policy, metadata, and chunk selection still need more work.

## Qualitative Reading By Stack

### specialist_supervisor

Strengths:

- best overall quality in this round;
- lowest average latency despite using the most semantically rich stack design;
- strongest public slice;
- strong restricted handling after the routing and Gemma-local planner fixes.

Weaknesses:

- still missed some keywords in admin+finance combination prompts;
- one protected academic comparison still over-clarified;
- one attendance follow-up fell back to academic-risk wording instead of staying purely attendance-centric;
- some public answers were correct but not as actionable as they could be.

Interpretation:

The specialist stack is now in the best overall position for product-facing quality, especially after the fixes that:

- stopped public shortcuts from stealing protected family panoramas;
- disabled fragile strict JSON tool mode for local Gemma;
- recovered recent-context follow-ups earlier in the routing flow.

### langgraph

Strengths:

- strong overall quality;
- solid public and protected performance;
- good restricted quality when the flow did not fail.

Weaknesses:

- two request failures still happened in this `50q`;
- some protected follow-ups still collapsed into clarification or generic boundary behavior;
- one attendance-detail question was answered with the wrong metric framing.

Interpretation:

LangGraph remains the strongest non-specialist competitor, especially when the state machine lands on the right branch. Its main remaining cost is operational: when it misses, it tends to miss through routing/exception behavior rather than just answer style.

### python_functions

Strengths:

- stable on most public and protected flows;
- best keyword pass among the non-specialist stacks in this round after the short 4Q validation;
- simple deterministic paths still make it predictable.

Weaknesses:

- weaker actionability on multi-document public prompts;
- still weaker than desired on restricted retrieval;
- some protected follow-ups still asked for clarification when the context should have been enough;
- latency was the highest average in this round.

Interpretation:

`python_functions` is still a very good baseline for deterministic service behavior, but it is paying for that simplicity in richer public synthesis and restricted retrieval depth.

### llamaindex

Strengths:

- competitive latency in some protected deterministic cases;
- public canonical-lane handling improved compared with earlier unstable phases.

Weaknesses:

- lowest overall quality in this round;
- five request failures;
- weakest restricted slice by a wide margin;
- multiple prompts still lost specificity or collapsed partially in multi-intent cases.

Interpretation:

LlamaIndex is no longer structurally broken, which is real progress, but it is still the least mature of the four stacks under this benchmark configuration. The main issue is no longer architecture alone; it is now robustness in the restricted and mixed-intent slices.

## Recurrent Failure Patterns

### 1. Restricted retrieval is still the biggest open problem

Evidence:

- automated recommendations point directly to restricted retrieval;
- `python_functions` restricted traces saturated `top_k` without becoming answerable;
- `llamaindex` restricted slice had multiple request failures and the worst quality.

Why this matters:

This is not just a retrieval-quality issue. It affects policy-sensitive product behavior and the trustworthiness of the assistant in high-value staff and internal-document scenarios.

### 2. Attendance-specific follow-ups still drift into general academic-risk answers

This appeared in more than one stack. The system now carries context much better than before, but attendance follow-ups can still be reinterpreted as generic academic-risk prompts when the wording is short and comparative.

### 3. Public multi-document answers still lose actionability

Some answers were factually acceptable but still too general. This did not look like hallucination. It looked more like “document-grounded but operationally weak synthesis.”

For TCC framing, this is important because it separates:

- factual correctness
- from decision usefulness

The benchmark shows that those are related but not identical dimensions.

## Threats To Validity

1. Protected slices used a real guardian Telegram context.

This was the correct choice for fairness in this repo, but it means the benchmark reflects a realistic local serving setup, not a fully synthetic auth-only harness.

2. Local Gemma serving is not the same as cloud production serving.

The benchmark is still valid for repository decisions, but latency and some answer-style behaviors are tied to the local runtime profile.

3. Keyword pass is useful but incomplete.

The manual review still matters because several low-keyword answers were directionally correct but insufficiently actionable, while some keyword-complete answers still had the wrong emphasis.

## Practical Conclusions

1. The current best overall stack in this repository is `specialist_supervisor`.
2. The best challenger is `langgraph`, especially when routing lands correctly.
3. `python_functions` remains the strongest deterministic baseline, but needs better restricted retrieval and richer public synthesis.
4. `llamaindex` is usable again, but still trails materially in reliability and restricted-document robustness.

## Next Priorities

1. Improve restricted retrieval metadata, query variants, and chunk selection.
2. Harmonize attendance follow-up handling across all stacks.
3. Increase actionability of public multi-document answers without sacrificing grounding.
4. Keep using the guardian-context-aware 50Q runner as the canonical local benchmark for future rounds.
